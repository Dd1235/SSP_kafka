## Why not a different language?


Kafka is written in Java, so standard benchmarks are too. However java suffers from JVM warmup. JIT is optimizing code for, and heavy JVM gc distorts result, for 30-40 sec benchmark like this. Go avoids this warm up phase

C/C++, would be incredibly fast and eliminate gc pauses, with tigher p99. However, writing complex lock free concurrent architecture (8 producers, 4 consumers, lag pollers, metric collectors all at once) without indroducing memory leaks or shared-state
would be extremely difficult

Rust: Excellent, zero-GC predictable latency as C++, but its strict compiler guarantees safe concurrenct. Maybe would show lower latency spikes than Go. Can rebuild the project in Rust! Steeper learning curve than Go.
Rust achieves memory safety through a unique system of ownership and borrowing, with a smart compiler that enforces strict rules about how data can be accessed and modified.

Go: Perfect balance. goroutines make spinning up so many concurrent workers trivial and readable. Go does have gc which adds slight latency pauces, but runtime/metrics package of go actively measure gc alongside the latency. 

Python has a GIL bottleneck. True multithreading isn't possible. It can simulate, but we need multiprocessing. Which introcues heavy IPC overhead. Sharing state like atomic counters across processes is slow and complex. 

JS because of the event loop. It's great at arync IO, but CPU bound tasks (which is what our benchmarking app does) will block the event loop.
