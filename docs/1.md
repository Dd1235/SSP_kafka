Why did snappy do poorly??

Snappy is famous for being fast and CPU efficient. Purely random bytes cannot be compression, random bytes into gzip or zstd would burn cpu cycles failing! Snappy is quick, gives up, and saves cpu on random data.

On our mixed payload, gzip and zstd can compress payload by 5x to 30x. Snappy prioritizing speed over size parely compresses at all. It's compression ratio is so poor, it pushes more bytes across the network, and makes broker do more disk I/O. in a heavy benchmark, dist time destroys latency advantage from fast CPU execution. 

zstd is highly advanced, dictionary based compression algo. to achieve its industry leading compressionr atios it has to maintain large sliding windows (dsa mentioned! maybe cp is not useful in real life tasks) and internal state buffers. Lots of memory scratchpad. Large batch -> large scratchpad buffers. Those buffers become garbage on compression. 

zstd would lifely dominate is written in Rust or CPP. wouldn't get 20ms to 50ms latency spikes. wo gc, pure cpu time? zstd heavily optimized for that!

throughput - bottlenecked by network bw and broker disk io. zstd and gzip achiteve higher compression ratios, so they reduce number of bytes sent over the wire and writtent to disk. 

snappy saturates the wire first.  You hit a hard throughput ceiling much earlier.

Snappy shines when bottlenecks are reversed, Network/Disk are abundant, and CPU is scare. 
The cluster: 10-node kafka cluster with 100 Gbps netowrk links and blazing fast NVMe SSD. 
The producer: Tiny cpu constrained kubernetes pod( only 0.5 CPU cores)

Or already compressed? like jpeg or encrypted blobs. zstd would waste cpu cycles on a meaningless task!

How do these work?

It's not profiling or instrumentation! It's like runtime internal accounting.

Go runtime is massive, heavily complex programs. When the gc decides to run, internal go runtime code executes. It knows when gc started and finished. All runtime does it update a few internal variables like `atomic.AddUInt64(&memstats.NumGC, 1)` in its own memory space.
When benchmark calls its, it isn't injection code or stopping the program to inspect state, simply asks go runtime to hand over values.


practically zero overhead. cost if only reading the data. In this bencmark, sysperf go routine wakes up every 2 seconds, reads a few integers from memory, and goes back to sleep. Less than amillisecond.

When you run `go build`, the go compiler takes your code, and the go runtime code, and mashes them into a `single statically linked executable binary`.

We have shared os threads, shared memoy space, and scheduler. Go runtime constantly multiplexes thousands of go routines. 

Go binary compile down to raw machine code specific to cpu and os, so its not wora ublike java where it compiles to bytecode.

but its very nice for cross compilation! compile annywhere for anywhere!

a real os thread requires 1mb to 4mb just to exist. A goroutine at just 2 kb. You can span 100k goroutiens wo crashing your laptop.

for a kafka benchmark or a standard web server, a 200 nano second scheduler overhead is invisible! For HFT thats eternity, so C++

In HFT, C++ engineers "pin" a specific thread to a specific physical CPU core and run it in an inifinite loop.

Entires CPU cache stays hot and OS never context switches the thread out.
