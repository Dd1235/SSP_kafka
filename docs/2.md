## 4 Consumer Lag

Most older benchmarks report consumer lag as a single number at the end of the run.
But a single number hides the dynamic. We measure lag as a polled time series. 

Goroutines wakes up every second, lag = hwm - commited_offest

We keep current as atomic int, for adaptive backpressure contorller. Backpressure controller lives over in hte producer pool and needs to check the current lag to decide whether to pauce producers. Producer pool - 10000times/sec, if we want mutex each time lock contention! Offets are committed asynchronously, saram consumer client auto commit its offset to the broker every second. 

10k messages/s is total aggregate rate. 

There is a 9th go routine in the producer pool called the `ThrottleController`. This is the brain. Reads the atomic lag, runs the hysteresis math, and decided is system is overloaded. If it is, flips atomic.Bool paused to true.

Each producer do zero math. Before sending, they call waitForCredit() that check if shared paused boolean.
An atomic boolean read from L1 CPU cache takes 2 to 10 ns. 

waitForCredit function is a `yielding sleep`, not a busy wait. The duration is half of the backpressure polling interval. If brain checks lag every 100 milliseconds, producers will sleep in chunks of 50 milliseconds. 

Parking: Go scheduler removes the program goroutine from the physical os threads and puts in to sleep on the global timer heap

Freeing the cpu: physical os thread is now totally free~

os timer: go registers a hardware timer with the os

waking time: os fires an interrupt. Go scheduler wakes up, grabs the producer goroutine and puts it back to runnable queue.

Wait, so there was no point of making all of these consumers!!!!!!!!!!!


Why not just use -consumers 1? If you used -consumers 1, all 12 partitions would be assigned to that single worker. Sarama would still spawn 12 ConsumeClaim goroutines, so your processing throughput would be identical!

However, you would have a major management bottleneck: Under the hood, that single worker has to run a background loop to constantly send "Heartbeats" to the broker to prove it is alive, and it has to manage fetching the 1MB data chunks from the network.

1 Partition = Exactly 1 Partition Goroutine. Because this topic has 12 partitions, there will always be exactly 12 Partition Goroutines doing the actual work, no matter how many Consumer Workers you spin up.

How the Math Works (The Rate)
Let's look at the true rate of a single Partition Goroutine. It reads a message, then calls time.Sleep(4ms). Because the OS scheduler isn't perfect, it usually sleeps for about 6ms.

Rate for ONE Partition: 1000ms / 6ms = ~166 messages per second.
Because there are exactly 12 partitions, there are exactly 12 of these goroutines running concurrently.

Total System Rate: 166 msg/sec × 12 partitions = ~1,992 messages per second.
(This is exactly why the benchmark results perfectly settled at 1,938 msg/s!)

If it's always 12 goroutines, why make 4 workers?
Here is what happens depending on what you set -consumers to:

-consumers 1: The broker assigns all 12 partitions to this single worker. The single worker spawns 12 Partition Goroutines.
-consumers 4 (The Default): The broker divides the partitions. Each of the 4 workers is assigned 3 partitions. Each worker spawns 3 Partition Goroutines. (Total = 12).
-consumers 12: The broker assigns 1 partition to each worker. Each worker spawns 1 Partition Goroutine. (Total = 12).
